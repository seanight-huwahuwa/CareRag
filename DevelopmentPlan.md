# グループホーム向け AIケアプラン作成支援システム 開発計画書

## 1. プロジェクト概要

### 開発コンセプト
**「入力は最小限、出力は専門的に」**
認知症対応型共同生活介護（グループホーム）の現場負担を軽減するため、利用者の状態を入力するだけで「施設サービス計画書（第2表）」の原案をAIが自動作成するWebアプリケーションを開発する。

### 独自の強み (RAG活用)
既存の優れたケアプラン事例サイト（tateyo-cm.com等）のデータを参考資料（ナレッジベース）として活用し、RAG（検索拡張生成）技術を用いることで、新人スタッフでもベテラン並みの専門的な表現や視点を取り入れたプラン作成を可能にする。

### ターゲットユーザー
*   グループホームの計画作成担当者
*   ホーム長・管理者
*   事務作業時間を短縮し、利用者ケアに時間を割きたい介護スタッフ

---

## 2. 機能要件 (Features)

システムはシンプルに **「入力・検索・生成・編集」** の4ステップで構成する。

### A. 入力機能 (Input)
*   **利用者基本情報:** 要介護度、年齢、認知症高齢者の日常生活自立度など。
*   **現状・課題キーワード:** フリーテキスト入力（例：「夜間の徘徊」「帰宅願望」「食事拒否」など）。
*   **意向（本人・家族）:** 本人がどう過ごしたいか、家族の願い。

### B. 検索機能 (Retrieval)
*   入力されたキーワードに基づき、データベース化された「立てよケアマネ」等の事例から類似性の高いケースを検索・抽出する。

### C. 生成機能 (Generation)
*   抽出した類似事例（参照データ）と入力情報をLLM（大規模言語モデル）に渡し、グループホームの理念（自立支援・共同生活）に沿った文章を作成する。

### D. 出力・編集機能 (Output)
*   生成されたテキストを画面上で修正可能にする。
*   ワンクリックでクリップボードにコピーし、業務ソフトやExcelに貼り付け可能にする。

## 3. 画面構成 (UI/UX デザイン)

開発効率とユーザビリティを考慮し、PC画面での利用を想定した **2カラム（左右分割）レイアウト** を採用する。

### 左パネル：入力エリア (Input Area)
ユーザーが情報を入力・操作するエリア。

1.  **基本属性選択:** ドロップダウン形式（要介護1〜5など）。
2.  **利用者のご様子 (Current Status):**
    *   テキストエリア。箇条書きやキーワードでの入力を推奨。
    *   *プレースホルダー：「例：トイレの場所がわからなくなる。他の利用者とトラブルになりやすい。」*
3.  **本人・家族の希望:** テキストエリア。
4.  **作成ボタン:** 「AIでプラン案を作成する」

### 右パネル：結果表示エリア (Output Area)
生成された「施設サービス計画書（第2表）」の構成要素をカード形式で表示する。**目標設定を明確にするため、長期・短期を別々のカードで表示する。**

1.  **【生活全般の解決すべき課題 (Needs)】**
    *   現状の問題点だけでなく、背景にある要因も含めた文章を表示。
    *   編集可能エリア。
2.  **【長期目標 (Long-term Goals)】**
    *   期間設定の目安と共に、目指すべき最終的な姿を表示。
    *   *UI表示例：「（期間：1年）ホームでの生活に慣れ、穏やかな気持ちで過ごすことができる」*
3.  **【短期目標 (Short-term Goals)】**
    *   長期目標を達成するための具体的なステップを表示。
    *   *UI表示例：「（期間：3ヶ月）職員の声かけにより、安心して入浴することができる」*
4.  **【援助内容 (Service Content)】**
    *   具体的な介助方法、声かけのタイミング、自立支援の視点を入れた内容。
5.  **【参考にした類似事例 (References)】**
    *   AIが生成の根拠とした元データの抜粋を小さく表示（信頼性の担保）。
6.  **【アクション】**
    *   「全項目をコピー」ボタン。

## 4. データパイプラインとRAG設計

信頼性の高いプランを作成するためのデータ処理フロー。

1.  **データ収集 (Data Collection):**
    *   Web上のケアプラン事例を収集。（※著作権に配慮し、学習用データとして内部処理のみに使用、または許可を得たデータを使用する）
2.  **データ加工 (Preprocessing & Chunking):**
    *   収集したデータを「ニーズ・目標・援助内容」のセットに構造化して保存する。
3.  **ベクトル化 (Embedding):**
    *   日本語に強いEmbeddingモデルを使用し、テキストをベクトルデータへ変換。
4.  **検索ロジック (Search Logic):**
    *   ユーザーの「現状・課題キーワード」と最も意味が近い事例をVector DBから数件ピックアップする。

## 5. 技術スタック (Tech Stack)

個人開発およびブログでの無料公開に適した構成。

*   **Frontend & Backend:** Python (Streamlit)
    *   *選定理由:* 素早いUI構築が可能で、データ処理との親和性が高い。
*   **LLM (AI Model):** OpenAI API (GPT-4o-mini)
    *   *選定理由:* 日本語性能が高く、かつAPI利用料が非常に安価であるため無料公開に適している。
*   **Vector Database:** ChromaDB (または Pinecone)
*   **Hosting:** Streamlit Community Cloud
    *   *選定理由:* HTTPS対応、GitHub連携での自動デプロイ、ブログへの埋め込み（iframe）が容易。

## 6. 開発ロードマップ

1.  **Phase 1: データセットの構築**
    *   事例データの収集とクリーニング。
    *   ベクトルデータベースの作成。
2.  **Phase 2: プロトタイプ開発 (Local)**
    *   Streamlitを用いた基本画面の実装。
    *   RAG検索精度のチューニング。
    *   プロンプトエンジニアリング（「〜すること」止め等の文体調整）。
3.  **Phase 3: デプロイとテスト**
    *   クラウド環境へのデプロイ。
    *   実際のケアマネジャー等の知人によるテスト利用とフィードバック。
4.  **Phase 4: ブログ公開**
    *   利用規約・免責事項の作成。
    *   ブログ記事への埋め込み公開。
